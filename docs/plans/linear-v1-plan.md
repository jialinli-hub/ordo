# Ordo 同款产品设计与实施计划（V1）

## 1. 背景与目标

本项目目标是构建一款面向软件研发团队的协作平台，覆盖 Ordo 的核心体验与高阶效率能力。系统需支持从需求拆解到迭代执行、代码关联、交付复盘的完整闭环，并具备后续企业化扩展能力。

### 1.1 业务目标

- 支持团队在系统内完整运行一个迭代（Cycle）。
- 提升任务管理效率与协作透明度。
- 建立自动化与代码托管平台（GitHub/GitLab）的联动能力。

### 1.2 成功指标（首版）

- 1 个试点团队连续 2 个迭代使用系统完成交付。
- Issue 的常见查询（列表筛选）P95 小于 300ms。
- 自动化规则与代码集成能力可稳定执行关键场景（如 PR 合并自动关闭 Issue）。

## 2. 用户角色与核心场景

### 2.1 用户角色

- Owner/Admin：组织配置、权限治理、自动化与集成管理。
- Project Lead：项目规划、里程碑管理、迭代安排、进度追踪。
- Developer/QA：处理任务、评论协作、状态更新、PR 关联。

### 2.2 核心场景

- 快速创建任务并按团队规范分类（优先级、标签、负责人）。
- 使用看板和列表推进任务状态流转。
- 在 Cycle 视角查看计划与完成情况。
- 从 PR、Commit 自动回链到 Issue 并触发状态更新。

## 3. 功能范围定义

### 3.1 P0（基础能力）

- Issue 管理：创建、编辑、状态流转、优先级、标签、负责人、截止时间。
- 项目管理：项目基本信息、里程碑、项目进度统计。
- Cycle：迭代配置、自动开启/关闭、周期统计。
- 视图能力：List/Board、筛选、排序、保存视图。
- 协作能力：评论、@提及、通知中心、我的任务。
- 报表能力：周期完成率、状态分布、个人任务概览。

### 3.2 P1（高阶能力）

- Roadmap：跨项目时间线与依赖关系展示。
- 自动化规则：触发器 + 条件 + 动作引擎。
- 集成能力：GitHub/GitLab 双向关联（PR/Commit/Branch）。
- 开放能力：Open API + Webhook。
- 模板能力：Issue 模板、Project 模板。
- 估点与排期：Story Points、迭代承载量管理。

### 3.3 P2（预留）

- 企业认证（SSO/SCIM）、审计日志增强、高级分析报表、数据迁移工具。
- 本文不纳入首版实施，仅保证架构可扩展。

## 4. 非功能要求（NFR）

- 性能：核心查询 P95 < 300ms；关键写操作 P95 < 500ms。
- 可靠性：关键 API 可用性 >= 99.9%。
- 安全：多租户隔离、RBAC、敏感数据安全存储。
- 可观测性：日志、指标、追踪、告警。
- 可扩展性：支持后续企业版能力平滑演进。

## 5. 技术架构建议

- 前端：React + TypeScript（后续支持命令面板与快捷键体系）。
- 后端：NestJS + Prisma + PostgreSQL。
- 异步任务：Redis + Queue（通知、自动化、Webhook 投递）。
- 集成层：Webhook 驱动 + 周期补偿任务。
- 部署：Docker + CI/CD（dev/staging/prod）。

## 6. 领域模型与数据设计

### 6.1 核心实体

- 组织与权限：`organizations`、`organization_members`、`project_members`
- 任务域：`projects`、`cycles`、`issues`、`issue_comments`
- 协作域：`issue_labels`、`notifications`、`activity_logs`

### 6.2 高阶实体

- 规划域：`roadmaps`、`roadmap_items`、`issue_relations`
- 自动化域：`automation_rules`、`automation_executions`
- 集成域：`integrations`、`external_links`、`webhook_endpoints`
- 效率域：`issue_templates`、`project_templates`、`issue_estimates`

### 6.3 关键设计约束

- 所有业务核心表必须携带 `organization_id`。
- Issue 编号按项目递增，需采用防并发方案。
- 状态流转受状态机约束，禁止非法跳转。
- 外部事件处理需幂等，避免重复执行。

## 7. 核心业务流程

### 7.1 任务闭环流程

1. 用户创建 Issue 并设定项目、优先级、负责人。
2. Issue 进入 Board/List，按状态推进。
3. 开发关联分支、提交、PR。
4. 集成模块同步 PR 状态并触发规则。
5. Issue 完成后进入 Cycle 复盘统计。

### 7.2 自动化流程示例

- 触发：PR merged。
- 条件：关联 Issue 当前状态为 `in_review`。
- 动作：更新 Issue 为 `done`，写入活动日志，通知关注者。

## 8. 里程碑计划（12 周）

### 8.1 阶段 0（第 1 周）准备期

- 完成需求冻结、字段字典与接口规范。
- 建立工程基线、CI、迁移与环境管理方案。

### 8.2 阶段 1（第 2-5 周）P0 闭环

- 第 2 周：组织、成员、项目、权限。
- 第 3 周：Issue CRUD、评论、活动流。
- 第 4 周：Cycle、列表/看板查询。
- 第 5 周：通知、我的任务、基础报表。

里程碑 M1：可支持试点团队完整跑一个迭代。

### 8.3 阶段 2（第 6-9 周）P1 高阶

- 第 6 周：Roadmap 与依赖关系。
- 第 7 周：自动化规则引擎。
- 第 8 周：GitHub/GitLab 集成。
- 第 9 周：Open API、Webhook、模板、估点。

里程碑 M2：具备接近 Ordo 的核心效率体验。

### 8.4 阶段 3（第 10-12 周）稳定与发布

- 性能优化、压测、慢查询治理。
- 安全加固、错误恢复、告警体系。
- UAT、灰度发布、正式上线。

里程碑 M3：可稳定对内外部团队使用。

## 9. 测试与验收

### 9.1 测试策略

- 单元测试：状态机、规则引擎、服务层。
- 集成测试：Issue 流程、集成回调、自动化链路。
- 端到端测试：创建任务到 PR 合并自动完结。
- 性能测试：Issue 查询与看板聚合场景。

### 9.2 验收清单（核心）

- 完成项目、Cycle、Issue 的创建与流转。
- 支持多维筛选与视图保存。
- 至少 2 条自动化规则可稳定执行。
- PR 可关联 Issue 并驱动状态更新。

## 10. 风险与应对

- 并发编号冲突：计数器表 + 事务锁。
- 自动化规则误触发：幂等键 + 执行频率限制 + 审计日志。
- 第三方集成不稳定：重试机制 + 死信队列 + 告警。
- 查询性能回退：索引治理 + SQL 观测 + 热点缓存。

## 11. 团队协作与执行机制

- 周会：每周一次里程碑检查，更新风险列表。
- 每日站会：同步阻塞与优先级调整。
- 评审机制：需求评审、技术评审、上线评审。
- 文档机制：接口变更、数据模型变更必须同步文档。

## 12. 部署实施补充

- Node + React 同项目统一部署方案详见：`docs/plans/node-react-unified-deployment.md`
- V1 阶段建议采用统一部署，优先保证交付速度与运维简洁性。
